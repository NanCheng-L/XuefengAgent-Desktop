# 版本公告生成

用户说"整理版本公告"时，按以下步骤执行：

## 步骤

1. **获取当前版本号**：从 `package.json` 读取当前版本号
2. **检查 CHANGELOG.md**：查找是否有当前版本的记录
3. **如果 CHANGELOG 有记录**：直接提取并格式化输出
4. **如果 CHANGELOG 没有记录**：
   - 运行 `git log --oneline` 获取提交历史
   - 找到上一个版本号的提交（格式如 `chore: 升级版本号到 X.Y.Z`）
   - 提取从上一个版本到现在所有提交的标题
   - 根据提交标题总结为版本公告
5. **格式化输出**：转换为 GitHub Release 公告格式

## Git 提交分类规则

根据提交信息前缀分类：
- `feat:` → 新增
- `fix:` → 修复
- `style:` → 改进（样式）
- `refactor:` → 改进（重构）
- `perf:` → 改进（性能）
- `docs:` → 文档
- `chore:` → 忽略（版本号、依赖等）
- `build:` → 忽略
- `ci:` → 忽略

## 输出格式

```markdown
## vX.Y.Z 版本公告

### 新增
- **功能名称**：功能描述

### 改进
- **改进名称**：改进描述

### 修复
- **修复名称**：修复描述

### 文档
- 文档变更描述
```

## 注意事项

- 版本号前加 `v`（如 `0.1.3` → `v0.1.3`）
- 只输出最新版本的内容，不包含历史版本
- 保持分类结构（新增/改进/修复/文档）
- 如果某个分类为空，不输出该分类标题
- 输出内容直接显示给用户，方便复制到 GitHub Release
- 如果 CHANGELOG 已有记录，优先使用 CHANGELOG 内容（更详细）
- 如果从 git log 生成，将提交标题改写为用户友好的描述
